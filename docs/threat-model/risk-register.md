# Risk Register

> **Example prompt:** *"Maintain the risk register. Input: stride-analysis.md, the attack trees, and any gaps surfaced while building. Phrase direct risks like the STRIDE analysis — 'An attacker could <action> in order to <goal>, leading to <impact>'; phrase process/coverage gaps as 'because <control> is missing, a flaw could be introduced or go unnoticed that an attacker exploits to <impact>'. Link each to a doomsday scenario. For each gap, ask me for the treatment decision (accept or mitigate) and the trigger for revisiting it; these are my calls, not yours. Output: a Markdown table with that risk statement plus a decision, rationale, and review trigger per row."*

The project's risks and their treatment: consciously accepted residual risks, and open risks with a
planned mitigation. Each has a rationale and a trigger that says when to revisit it. Most deferrals
share one trigger: **before any production deployment**.

**Risks are not yet rated.** Likelihood, impact, and a resulting priority are still to do (see the
threat-model [backlog](README.md#not-yet-modeled-backlog)). For now ordering is rough and by hand, and
each risk links to the doomsday scenario it would cause.

| ID | Risk | Decision | Rationale | Review trigger |
|----|------|----------|-----------|----------------|
| RR-1 | Because we rely on django-allauth's upstream tests rather than our own, a defect or misconfiguration in the auth flows (registration, email verification, login/logout) could go unnoticed and let an attacker take over an account (FR-1, SR-3, part of FR-2 → D1). | Accept (this iteration) | Provided and tested upstream by django-allauth, which is enough for a local first pass. These flows are the main account-takeover path, so we will add our own tests for them before production. | Before production deployment; sooner if we customize the signup, verification, or login flow. |
| RR-2 | With no proper SAST in place (only a basic linter), a developer could introduce an injection or authorization flaw that goes uncaught and that an attacker exploits to read data or run code (→ D1/D2). | Open — mitigate | For now the linter plus negative access-control tests and manual review cover the gap; a proper, Django/DRF-aware SAST tool with custom rules is specified ([testing-strategy.md](../requirements/testing-strategy.md)) but not built. Even once built, the deepest authz semantics stay on tests + review. | Before production, or when CI is set up. |
| RR-3 | An attacker or runaway client could flood the API with expensive or high-volume requests (no rate limiting or quotas) to exhaust resources and make the service unavailable (STRIDE DoS → D3). | Accept (this iteration) | Local, debug-only basis; not publicly exposed. | Before any production / public deployment. |
| RR-4 | With no audit log of user/admin actions, an attacker's activity (or a breach) could go undetected and unattributable, so it cannot be reconstructed or scoped (STRIDE Repudiation → amplifies D1/D2). | Accept (this iteration) | Out of MVP scope; not needed for a local basis. | Before production deployment, or if compliance requires it. |
| RR-5 | If deployed without hardening (no TLS/HSTS, `DEBUG` on, weak secret handling), an attacker could intercept traffic or read leaked internal detail and credentials (SR-4, SR-11 → D1). | Accept (local-only) | Running locally in debug by design; currently avoided only by not deploying. | Before any non-local / production deployment. |
| RR-6 | Without an ASVS v5 / Django deployment-checklist review, a required control could be missing unnoticed, leaving a gap an attacker could exploit (→ D1/D2). | Accept (this iteration) | Threat-model-driven controls came first; a standards pass is later work. | Before production deployment. |
| RR-7 | An attacker with a stolen, phished, or guessed password could authenticate as the user (MFA available but not enforced) and take over the account (B1 Spoofing → D1). | Open — mitigate | Enforcement is an auth-hardening decision deferred past the MVP core. | Before production, or when signup opens to untrusted users. |
| RR-8 | An attacker could brute-force or credential-stuff the API and token endpoints (no throttling or lockout) to take over accounts (B1 Spoofing → D1). | Open — mitigate | Default allauth limits cover the login view; the API/token paths are unthrottled. | Before production / public exposure. |
| RR-9 | An attacker could slip a tampered dependency or build artifact into the system (no SBOM, commit/artifact signing) and have it run undetected, leading to code execution or data theft (B2 Repudiation → D2). | Open — mitigate | Dependencies are hash-pinned via `uv.lock`; provenance and signing are not in place. | Before production, or when a release pipeline exists. |
| RR-10 | An attacker could push a changed or poisoned image under the same tag (base images pinned by tag, not digest) and have it pulled into a build, running attacker-controlled code (B2 Tampering → D2). | Open — mitigate | Tag pinning is mutable; no image verification. | Before production. |
| RR-11 | An attacker could get a malicious or typosquatted package added and trusted (no dependency source policy beyond hashes), compromising the build (B2 Spoofing → D2). | Open — mitigate | `uv.lock` verifies integrity but there is no source allowlist or policy. | Before production. |
| RR-12 | An attacker who obtains pipeline credentials or abuses an over-privileged job could push malicious code to production (no least-privilege CI/CD) (B2 Elevation / DoS → D2). | Open — mitigate | No CI/CD pipeline exists yet. | When a CI/CD pipeline is introduced. |
| RR-13 | Because the compliance groundwork is not in place (privacy notice, lawful basis, data-subject-rights tooling, breach process, processor agreements, retention), processing real personal data could violate GDPR / UK GDPR / US state privacy laws, leading to regulatory fines and legal action — a data breach (D1) would compound it. See [compliance.md](../governance/compliance.md). | Open — mitigate | Currently local/debug with no real users; the landscape and gaps are mapped in [compliance.md](../governance/compliance.md). The work must be completed and reviewed by a qualified professional before launch. | Before processing any real personal data (before production). |

Each risk is revisited at its trigger; none is permanent.
